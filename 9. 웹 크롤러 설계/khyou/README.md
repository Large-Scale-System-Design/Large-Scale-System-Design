# 웹 크롤러
웹 크롤러는 검색 엔진에서 널리 쓰는 기술로, 웹에 새로 올라오거나 갱신된 콘텐츠를 찾아내는 것이 주 목적이다.
- 콘텐츠: 웹 페이지, 이미지, 비디오, PDF 등

크롤러는 다양하게 이용된다.
- 검색 엔진 인덱싱: 구글 검색 엔진이 사용하는 웹 크롤러이다. 웹 페이지를 모아 **검색 엔진**을 위한 로컬 인덱스를 만든다.
- 웹 아카이빙: 나중에 사용할 목적으로 장기보관하기 위해 웹에서 정보를 모으는 절차
- 웹 마이닝: 웹을 통해 유용한 지식들을 채굴한다. 유명 금융 기업들은 크롤러를 사용해 주주총회 자료나 연차 보고서를 다운받아 기업의 핵심 사업 방향을 알아내기도 한다.
- 웹 모니터링: 인터넷에서 저작권이나 상표권이 침해되는 사례를 모니터링한다.

# 1단계 문제 이해 및 설계 범위 확정

웹 크롤러의 기본 알고리즘은 간단하다.
1. URL 집합 중 하나를 꺼내서 URL들이 가리키는 모든 웹 페이지를 다운로드한다.
2. 웹 페이지에서 URL을 추출한다.
3. 추출된 URL들을 다운로드할 URL 목록에 추가하고 위의 과정을(1 - 2 - 3) 반복한다.

웹 크롤러 알고리즘은 간단하지만 웹 크롤러가 이렇게 단순하게 동작하지는 않는다.
그러니 설계를 진행하기 전에 질문을 던져서 요구사항을 알아내고 설계 범위를 좁혀야 한다.
웹 크롤러가 만족 시켜야 할 속성에 대해 살펴보자.
- 규모 확장성: 웹은 방대하다. 따라서 병행성을 활용하면 보다 효과적일 것 이다.
- 안정성: 비정상적 입력이나 환경에  잘 대응할 수 있어야 한다. (ex. 악성 코드)
- 예절: 짧은 시간 동안 너무 많은 요청을 보내서는 안된다.
- 확장성: 새로운 형태의 콘텐츠를 지원하기가 쉬어야 한다.

# 2단계 개략적 설계안 제시 및 동의 구하기

1. **시작 URL 집합**
- 시작 URL 집합은 원칙적으로 **해당 도메인에 속한 모든 페이지의 URL**을 후보로 삼을 수 있다.
- 다만 실제 크롤링에서는 전체 URL 공간이 매우 크기 때문에, **URL 공간을 여러 작은 부분집합으로 나누는 전략**을 사용한다.
	- 예를 들어 디시인사이드의 경우 /board/, /gallery/, /major/와 같은 상위 경로 단위로 URL 공간을 분리하고, 각 경로의 목록 페이지를 시작 URL로 선택할 수 있다.
- 중요한 점은 **어떤 URL을 시작점으로 삼아야 한다는 정답은 존재하지 않는다**는 것이다.
	- 따라서 크롤러의 목적(최신 글 수집, 특정 게시판 분석, 전체 구조 파악 등)에 따라 시작 URL을 설계하면 된다.

2. **미수집 URL 저장소**
- 현대적 웹 크롤러는 크롤링 상태를 두 가지로 관리한다.
	1. 다운로드할 URL 
	2. 다운로드된 URL
- 다운로드할 URL을 저장 관리하는 컴포넌트를 미수집 URL 저장소라고 부른다.
	- FIFO, 큐라고 생각하면 된다.

3. **HTML 다운로더**
- 인터넷에서 웹 페이지를 다운로드하는 컴포넌트

4. **도메인 이름 변환기**
- 웹 페이지를 다운받으려면 URL을 IP 주소로 변환하는 절차가 필요하다.
- HTML 다운로더는 도메인 이름 변환기를 사용하여 IP 주소를 알아낸다.

5. **콘텐츠 파서**
- 웹 페이즈를 다운로드하면 파싱과 검증 절차를 걸쳐야 한다. 이상한 웹 페이지는 문제도 되고, 저장 공간을 차지하기 때문

6. **중복 콘텐츠인가?**
- 최근 연구에 따르면 29% 가량의  웹 페이지 콘텐츠는 중복이다.
- 이미 시스템에 저장된 콘텐츠임을 알아낼 수 있는 좋은 방법은 웹 페이지의 해시 값을 비교하는 것이다.

7. **콘텐츠 저장소**
- HTML 문서를 보관하는 시스템
- 데이터 양이 너무 많으므로 대부분의 콘텐츠는 디스크에 저장해야 한다.
- 인기 있는 콘텐츠는 메모리에 두어 접근 지연시간을 줄일 수 있다.

8. **URL 추출기**
- HTML 페이지를 파싱하여 링크들을 골라내는 역할을 한다.
- 상대 경로는 전부 절대 경로로 변환해야 특정이 된다.

9. **URL 필터**
- 블랙리스트에 포함된 URL 등을 크롤링 대상에서 배제하는 역할을 한다.
	- 블랙리스트는 따로 있는 건 아니고, 필요 없는 확장자거나 접속시 오류가 나는 URL 등을 뜻함

10. **이미 방문한 URL?**
- 이미 방문한 적이 있는 URL인지 추적하면 같은 URL을 여러 번 처리하는 일을 방지할 수 있어서 서버 부하를 줄이고 시스템이 무한 루프에 빠지는 일을 방지할 수 있다.

11. URL 저장소
- 이미 방문한 URL을 보관하는 저장소

## 웹 크롤러 작업 흐름
1. 시작 URL들을 미수집 URL 저장소에 저장한다.
2. HTML 다운로더는 미수집 URL 저장소에서 URL 목록을 가져온다.
3. HTML 다운로더는 도메인 이름 변환기를 사용하여 URL의 IP 주소를 알아내고, 해당 IP 주소로 접속하여 웹 페이지를 다운받는다.
4. 콘텐츠 파서는 다운된 HTML 페이지를 파싱하여 올바른 형식을 갖춘 페이지인지 검증한다.
5. 콘텐츠 파싱과 검증이 끝나면 중복 콘텐츠인지 확인하는 절차를 개시한다.
6. 중복 콘텐츠인지 확인하기 위해서, 해당 페이지가 이미 저장소에 있는지 본다.
	1. 이미 저장소에 있는 콘텐츠인 경우에는 처리하지 않고 버린다.
	2. 저장소에 없는 콘텐츠인 경우에는 저장소에 저장한 뒤 URL 추출기로 전달한다.
7. URL 추출기는 해당 HTML 페이지에서 링크를 골라낸다.
8. 골라낸 링크를 URL 필터로 전달한다.
9. 필터링이 끝나고 남은 URL만 중복 URL 판별 단계로 전달한다.
10. 이미 처리한 URL인지 확인하기 위하여, URL 저장소에 보관된 URL인지 살핀다. 이미 저장소에 있는 URL은 버린다.
11. 저장소에 없는 URL은 URL 저장소에 저장할 뿐 아니라 미수집 URL 저장소에도 전달한다.

# 3단계 상세 설계
## DFS를 쓸 것인가, BFS를 쓸 것인가

> 웹 크롤러는 **특정 페이지를 깊게 파고드는 것보다, 가능한 한 많은 페이지를 고르게 수집하는 것**이 목적이다. DFS는 한 경로를 끝까지 따라가므로 일부 영역만 과도하게 크롤링하고, 다른 영역은 아예 방문하지 못할 수 있다. 반면 BFS는 시작 URL과 **가까운 페이지부터 레벨 단위로 확장**하므로 전체 사이트를 균형 있게 탐색할 수 있다. 또한 BFS는 최신 글·핵심 페이지처럼 **중요도가 높은 상위 레벨 페이지를 빠르게 확보**할 수 있다.
> 이 때문에 실무 크롤러는 기본 탐색 전략으로 BFS를 선택한다.

## 왜 우선순위(Priority)가 필요한가

> 현실적인 크롤링 환경에서는 **모든 URL을 동일하게 다룰 수 없다**. 서버 부하, 시간 제한, 저장 공간 등의 제약으로 인해 **어떤 URL을 먼저 방문할지 결정해야 한다**. 이때 단순한 BFS 큐 대신 **우선순위 큐**를 사용해 중요도가 높은 URL을 먼저 크롤링한다. 예를 들어 목록 페이지 > 게시글 페이지, 최신 글 > 오래된 글, 트래픽 높은 게시판 > 비활성 게시판처럼 우선순위를 줄 수 있다. 즉, 우선순위는 크롤러의 **목적을 탐색 순서에 반영하기 위한 장치**다.

## 미수집 URL 저장소
미수집 저장소를 잘 구현하면 예의를 갖춘 크롤러, URL 사이의 우선순위와 신선도를 구별하는 크롤러를 구현할 수 있다. 미수집 URL 저장소의 구현 방법에 대해 알아보자

### 예의
- 웹 크롤러는 수집 대상 서버로 짧은 시간 안에 너무 많은 요청을 보내는 것을 삼가해야 한다. 대상 서버를 다운 시키거나 공격으로 취급당할 수 있다.
- 그렇기 때문에 동일 웹 사이트에 대해서는 한 번에 한 페이지만 요청해야 한다.
	- 스레드 여러개가 병렬로 한 사이트를 동시에 요청을 보내는 것이 아니라, 수집 대상 서버에 스레드 하나 씩 할당

#### 예의를 고려한 설계
- 큐 라우터: 같은 호스트에 속한 URL은 언제나 같은 큐에 가도록 보장하는 역할
- 매핑 테이블: 호스트 이름과 큐 사이의 관계를 보관하는 테이블
- 큐 선택기: 큐 선택기는 큐들을 순회하면서 큐에서 URL을 꺼내서 해당 큐에서 나온 URL을 다운로드하도록 지정된 작업 스레드에 전달하는 역할을 한다.
- 작업 스레드: 전달된 URL을 다운로드하는 작업을 수행한다.

### 우선순위를 고려한 설계
디시인사이드를 예로 들면 디시인사이드 메인 홈페이지보다 디시 회사 소개 홈페이지가 더 중요도를 갖는다고 보기는 어렵다. 그렇기 때문에 우선순위를 고려해야 한다.

유용성에 따라 URL의 우선순위를 나눌 때는 페이지랭크, 트래픽 양 등 다양한 척도를 사용할 수 있다.
- 순위결정장치: URL을 입력으로 받아 우선순위를 계산한다.
- 큐: 우선순위별로 큐가 하나씩 할당된다.
- 큐 선택기: 임의 큐에서 처리할 URL을 꺼내는 역할을 담당한다. 순위가 높은 큐에서 더 자주 꺼내도록 프로그램되어 있다.

### 신선도
웹 페이지는 수시로 추가되고, 삭제되고, 변경된다. 따라서 데이터의 신선함을 유지하기 위해서는 이미 다운로드한 페이지라고 해도 주기적으로 재수집할 필요가 있다.

## 미수집 URL 저장소를 위한 지속성 저장장치
검색 엔진을 위한 크롤러의 경우, 처리해야 하는 URL의 수는 수억 개에 달한다. 그러니 그 모두를 메모리에 보관하는 것은 안정성이나 규모 확장성 측면에서 좋지 않다. 그렇다고 디스크에 저장하면 병목지점이 된다. 따라서 절충안을 택한다. 대부분의 URL은 디스크에 두지만 IO 비용을 줄이기 위해 메모리 버퍼에 큐를 둔다. 버퍼에 있는 데이터는 주기적으로 디스크에 기록한다.

## HTML 다운로더
### Robots.txt
- 웹사이트가 크롤러와 소통하는 표준적 방법이다.
- 이 파일에는 크롤러가 수집해도 되는 페이지 목록이 들어 있다.
- ex) https://www.amazon.com/robots.txt, https://www.coupang.com/robots.txt
- robots.txt는 보안 기능이 아니라 단지 크롤러에게 알려주기 위한 용도의 파일이다. 악성 봇은 무시하고 크롤링할 수 있다.

### 성능 최적화
1. 분산 크롤링: 크롤링 작업을 여러 서버에 분산하는 방법. 각 서버는 여러 스레드를 돌려 다운로드 작업을 처리한다.
2. 도메인 이름 변환 결과 캐시: DNS 요청을 보내고 결과를 받는 작업의 동기적 특성 때문에 병목 지점이 된다. 그렇기 때문에 DNS 조회를 통해 얻어진 도메인 이름과 IP 주소를 캐시에 보관해두고 주기적으로 갱신하면 성능을 최적화할 수 있다.
3. 짧은 타임아웃: 웹 서버는 응답이 느리거나 아예 응답을 하지 않을 수 있기 때문에 짧은 타임아웃을 설정하여넘어가도록 한다.
4. 지역성: 크롤링 서버를 지역별로 분산하여 통신 시간을 줄인다.

### 안정성
- 크롤링 상태 및 수집 데이터 저장: 장애가 발생한 경우에도 쉽게 복구할 수 있도록 상태 및 수집 데이터를 저장한다.
- 예외 처리: 예외가 발생해도 전체 시스템이 중단되는 일 없이 그 작업을 우아하게 이어나갈 수 있어야 한다.

### 확장성
새로운 형태의 콘텐츠를 쉽게 지원할 수 있도록 신경 써야 한다. 새로운 모듈을 끼워 넣음으로써 새로운 형태의 콘텐츠를 지원할 수 있도록 설계가 필요하다.

### 문제 있는 콘텐츠 감지 및 회피
1. 중복 콘텐츠: 해시나 체크섬을 사용하여 중복 콘텐츠를 걸러내어야 한다.
2. 거미 덫: 크롤러를 무한 루프에 빠뜨리도록 설계한 웹 페이지이다. 파훼하는 알고리즘을 만들기 어려워서 사람의 개입이 필요하다.
3. 데이터 노이즈: 광고나 스크립트 코드와 같은 크롤링하는 목적에 맞지 않은 의미 없는 데이터는 제외해야 한다.

# 추가
## 서버 사이드 렌더링
- 이커머스 등 많은 웹 서비스는 검색 엔진 노출이 중요하다. 클라이언트 사이드 렌더링은 초기 HTML이 비어 있고 JavaScript 로 동적으로 페이지를 구성하기 때문에, 일부 검색 엔진 봇이 콘텐츠를 정확히 인식하기 어렵다. 이에 따라 서버에서 HTML을 완성해 전송하는 서버 사이드 렌더링(SSR)이 다시 주목받고 있다. (예: CSR – React, SSR – Next.js)
